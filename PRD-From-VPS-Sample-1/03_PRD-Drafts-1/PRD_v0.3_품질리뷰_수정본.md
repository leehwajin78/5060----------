## 2. 챕터/파트별 수정 방향

1. **§7. 사용자 스토리와 수용 기준(AC)**
   - 예외 상황(API 응답 지연, LLM 환각, 데이터 매핑 실패)에 대한 **Negative AC**를 Given-When-Then 구조로 추가하여 검증 가능성 확보.
2. **§8. 기능 요구사항(Functional)**
   - 핵심 기능에 대안(인간 코치, 크몽 외주) 대비 **정량적 비교 수치(비용, 시간, 정확도 등)** 추가.
3. **§9. 비기능 요구사항(NFR)**
   - 시스템 성능(API 응답 속도 < N초), 가용성(99.9%), **모니터링 알림 기준**(에러율 임계치, Slack 연동 등) 수치화 명문화.
4. **§14. 근거 (Proof)**
   - 근거 항목이 어떤 **실험(E1~E8)**으로 검증되는지 매핑하는 추적성(Traceability) 컬럼을 추가하여 테스트와 근거를 직접 연결.
